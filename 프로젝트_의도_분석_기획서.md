# 돈이와 함께한 모든 순간들: 프로젝트 의도 분석 기획서

## 1) 프로젝트 한 줄 정의
- 감정 태깅 기반의 소비 기록을 통해, 사용자가 지출을 죄책감이 아닌 "나를 위한 시간 기록"으로 인식하게 만드는 모바일형 개인 재무 다이어리.

## 2) 기획 의도(추정)
- 이 프로젝트는 단순 가계부가 아니라, 소비를 "추억/경험/성장"의 맥락으로 재해석하려는 의도가 강함.
- 캐릭터(돈이)와 대화형 UI를 통해 기록 진입장벽을 낮추고, 월별 목표/고정비 관리와 결합해 생활 재무 루틴을 형성하려는 목적이 보임.
- 핵심 메시지:
  - "단순한 소비를 넘어, 가치 있는 순간을 기록하세요."
  - "Don't Worry, Be Happy"

## 3) 타깃 사용자
- 전통적인 숫자 중심 가계부가 부담스러운 사용자
- 특히 mbti가 infp인 2030 청년 사용자
- 감정/라이프로그 중심으로 지출을 회고하고 싶은 사용자
- 월 단위 예산(수입/고정비/가용금액)과 지출 기록을 함께 관리하려는 개인

## 4) 핵심 사용자 문제
- 지출 기록이 귀찮고 지속되지 않음
- 돈 관리를 하면 죄책감/압박감이 먼저 듦
- 월별 계획(수입/고정비/목표)과 실제 사용 내역이 분리되어 관리됨

## 5) 제품이 제공하는 해결 방식
- 캐릭터 챗 인터페이스로 기록 경험을 친근화
- 태그(성장/행복/생활/방황)로 지출의 의미를 분류
- 월별 설정(수입/고정비/목표) + 실지출을 연결해 `잔액(Current Balance)`을 즉시 피드백
- 통계 화면에서 태그 비중/월 지출 히스토리를 시각화해 회고를 유도

## 6) 핵심 UX 구조
1. 로그인형 인트로
- "기록하러 가기"로 앱 진입
- 구글로 로그인하기를 통해 간편한 로그인

2. HOME
- 월 이동(이전/다음)
- 캐릭터 응원 문구 랜덤 노출
- 현재 잔액 표시
- "이번 달 약속 정하기" 또는 기존 목표 확인

3. CHAT
- 지출 기록 생성/수정/삭제
- 지출 1건당 캐릭터 반응 메시지 제공
- 신규 월 진입 시 전월 설정 복사 제안

4. MEMORY(통계)
- 태그별 소비 비중(%)
- 월별 소비 메시지(최대 비중 태그 기반)
- 상세 기록 펼침/접기
- 바 차트(현재 월만 값 표시되는 연간 프레임)

5. SETTING
- 수입 상세 항목 관리
- 고정비 상세 항목 관리
- 가용금액(수입-고정비) 표시
- 월 목표 저장

## 7) 데이터 모델(로컬 저장)
- 저장 방식: `localStorage`
- 키:
  - `doni_test_exp_v2`: 지출 기록 배열
  - `doni_test_set_v2`: 월별 설정 객체
- 지출 레코드:
  - `id, year, month, fullDate, item, cost, tag, customHash`
- 월 설정 레코드:
  - `inc, fix, incDetails[], fixDetails[], goal`

## 8) 화면/브랜드 톤앤매너
- 감성 키워드: 따뜻함, 위로, 일상성, 부담 완화
- 시각 방향: 모바일 앱형 카드 UI, 라운드, 저채도 웜톤
- 카피 전략: "관리"보다 "약속/대화/추억" 중심 언어

## 9) 현재 구현 기준 제품 단계 판단
- 단일 `index.html` 기반의 프로토타입/PoC에 가까움
- 백엔드/계정 시스템 없이 로컬 상태 저장 중심
- 개인 사용 및 UX 검증 단계에 적합

## 10) 기획 관점의 강점
- 차별점이 명확함: "감정 기반 재무 기록"
- 지속 사용을 유도하는 캐릭터-대화 구조
- 월 예산 설정과 소비 회고가 한 흐름으로 연결됨

## 11) 기획 관점의 한계/리스크
- 데이터 영속성 한계: 브라우저/기기 변경 시 데이터 이관 어려움
- 통계 정교성 한계: 연간 차트는 현재 월만 반영
- 다중 사용자/동기화/보안 고려 미흡
- 태그 기반 인사이트 확장(카테고리/목표 달성률/추세 분석) 여지 큼

## 12) 다음 기획 우선순위 제안
1. 계정/동기화 도입(데이터 보존)
2. 통계 고도화(월별 추세, 목표 달성률, 고정비 대비 변동비 분석)
3. 리마인더/습관 루프(기록 유도 알림, 주간 회고)
4. 캐릭터 피드백 개인화(사용 패턴 기반 메시지)

이 문서는 코드 구조와 UI 카피를 기반으로 "의도"를 추론한 분석 기획서입니다.

## 13) 최종 구현 전 필수 보강 컨텍스트 (누락 항목 도출)
아래는 "개발 가능한 수준"으로 기획서를 끌어올리기 위해 반드시 채워야 하는 항목들입니다.

### A. 제품 범위/릴리즈 전략
- MVP 범위 확정:
  - 이번 릴리즈에서 반드시 제공할 기능/제외할 기능(Out of Scope) 명시

- 플랫폼 범위:
  - 웹앱(PWA)만 할지, 네이티브(iOS/Android) 동시 추진할지

- 릴리즈 단위:
  - Alpha(내부), Beta(소수 사용자), Public 출시 기준 정의

A. 제품 범위/릴리즈 전략 답변: 
  - 다중 사용자, 동기화, 보안은 최소한으로 구성함.(추후 적용할 예정)
  - 필수기능: 구글 기반 회원가입, 로그인(일반 로그인, 회원가입은 지원안함), 화면에 연동된 기능들(기록 저장, 불러오기 등)
  - 웹앱 기반.
  - 추후 ios 연동 예정
  - 릴리즈 단위는 퍼블릭이나, 소수 사용자가 사용할 예정임.

### B. 사용자/시나리오 명세
- 핵심 페르소나 1~2개 구체화:
  - 나이대, 기록 습관, 재무 고민, 앱 사용 맥락
- 대표 사용자 여정(시나리오) 문서화:
  - 첫 진입 -> 월 설정 -> 기록 -> 월말 회고까지
- 실패 시나리오 정의:
  - 기록 중단, 목표 미달, 데이터 유실 상황에서의 UX 대응

B. 사용자/시나리오 명세 답변:
- 핵심 페르소나 구체화: 
30대 여성, 기록은 하고싶은데 귀찮은 성향. 귀엽고 아기자기한 감성 좋아함. 수입과 지출이 있는 직장인임. 지출기록을 적으면서 지출에 대한 감정을 기록할 수 있으며, 지출에 대한 의미부여를 할 수 있게 한다. 심플한 일기와 같음.

- 사용자 여정: 
첫진입 시 소개 페이지 확인 후 화면 내 버튼 통해 구글로 회원가입 후 구글로그인으로 간편하게 진입함. 셋팅 탭에서 고정수입과 고정지출계획을 적고 그달의 목표를 적는다. 채팅 탭에서 '돈이'와 대화하는 방식으로 지출기록을 할 수 있다. 지출할 때 행복, 성장, 생활, 방황의 4가지 카테고리로 분류함. 어떤 카테고리의 소비를 많이 했는지 메모리탭에서 확인 가능.

### C. 기능 요구사항 상세화 (화면별)
- HOME:
  - 잔액 계산식 확정(소수점/음수 표시 정책 포함)
- CHAT:
  - 태그 미선택/금액 0원/미래 날짜 입력 등 예외처리 정책
  - 수정/삭제 권한 범위(과거 월 기록 수정 허용 범위)
- MEMORY:
  - 통계 기간 기준(해당 월만/누적/사용자 선택)
  - 차트 데이터 정의(현재는 당월만 반영되는 구조 개선 여부)
- SETTING:
  - 전월 복사 시 어떤 값을 복사할지(수입/고정비/목표/상세항목)
  - 월 중간 설정 변경 시 과거 기록 반영 정책

C. 기능 요구사항 상세화 답변:
- home: 원화 표시, 음수 표시, 소수점은 없음. 터치형 말풍선으로 소비에 관한 메시지 노출

- chat: 태그 미선택, 금액 0원, 미래날짜 입력 발생 허용 안함. 시도 시 불가하다는 메시지 노출하도록함.

- memory: 일단 통계 기간은 해당 월만, 차트 데이터는 데이터가 있는 전체 월이 다 보이도록 하되 현재 월에는 색상 더 진하게 표시 

### D. 정보 구조/데이터 정책
- 데이터 스키마 버전 전략:
  - 향후 필드 추가 시 마이그레이션 규칙
- 삭제 정책:
  - 하드 삭제 vs 소프트 삭제(복구 기능) 선택
- 날짜/타임존 정책:
  - 사용자 로컬 타임존 고정 여부
- 숫자/통화 정책:
  - 통화 단위(KRW 고정 여부), 천 단위/반올림 규칙

D. 정보 구조/데이터 정책 답변: 
- 향후 필드 추가 시 마이그레이션은 tmp 테이블 생성 후 복제하는 방식으로 진행하여 데이터 누락방지
- 깃으로 형상관리할 예정이므로 하드삭제
- 타임존은 대한민국 서울
- 통화단위는 일단 krw 고정. 반올림은 없음(소수점 발생하지 않음)


### F. 비기능 요구사항
- 성능:
  - 기록 N건에서 렌더링 목표시간(예: 1,000건 기준 1초 이내)
- 접근성:
  - 대비, 폰트 크기, 스크린리더 대체 텍스트 기준
- 신뢰성:
  - 저장 실패/브라우저 스토리지 제한 시 복구 UX
- 보안/프라이버시:
  - 민감정보 정의, 저장 암호화 필요성, 로그 정책

F. 비기능 요구사항 답변:
- 성능: 1000건 기준 1초 이내
- 접근성: 일단 고려안함
- 신뢰성: 저장실패, 스토리지 제한 시 사유밝히고 오류팝업 띄운 후 고객고지하는 ui생성
- 보안 프라이버시 일단 고려 안함.  

### G. 기술/아키텍처 결정 포인트
- 프론트엔드 구조:
  - 단일 HTML 유지 vs 프레임워크 전환(React/Vue/Svelte)
- 상태관리:
  - 로컬 상태 + 스토리지 추상화 계층 도입 여부
- 백엔드 도입 시점:
  - 인증/동기화/백업을 ㄴ언제 붙일지
- 분석 도구:
  - 사용자 행동 이벤트 수집(Amplitude/GA 등) 여부


G. 기술/아키텍처 결정 포인트 답변:
- 프론트앤드 구조: 프레임워크 전환 예정(vue.js)
- 상태관리용 관리자 페이지 별도 포트, 화면으로 구성
- 백앤드 도입 시점: 지금부터 고려해야함. 수파베이스로 백앤드 구성하기 바람. 이 프로젝트의 모든 자원은 도커로 구성하여 나중에 배포하기 쉽게 해야 함. 
- 분석도구 당장은 고려안함
